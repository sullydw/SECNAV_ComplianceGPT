## 1. Purpose

Phase D defines an append-only pending global rule candidate log for corrections classified as `possible_secnav_manual_rule` or `bug_validator_gap`. The log captures structured, sanitized candidate records for future human or AI-assisted review. It does **not** implement promotion to global rules, modify validators, or alter the renderer. Candidate logging is a prerequisite for future Phase E review/promotion but is intentionally limited to writing and managing a local-only, gitignored record store.

---

## 2. Eligible Corrections for Pending Candidate Logging

Only corrections meeting **all** of the following criteria are eligible:

| Criterion | Requirement |
|---|---|
| **Classification** | `correction_type` is `"possible_secnav_manual_rule"` or `"bug_validator_gap"` |
| **Scope** | Originally captured with `scope="current_session"` |
| **User approval** | Explicit user confirmation received via two-step approval workflow (see Section 4) |
| **No validator conflict** | The correction did **not** trigger `validator_conflict=True` |
| **Not previously rejected** | The session correction is not soft-marked `user_rejected=True` |
| **Value stability** | The corrected value is non-empty and not a placeholder like "TBD" or "TODO" |
| **Session recency** | Correction belongs to the current active session only |
| **Not already logged** | The same `(field_path, corrected_value, doc_type, component)` fingerprint is not already present with `status="pending"` or `"under_review"` |
| **Field path whitelist** | `field_path` is **not** a `body.*` path. Body text is document-specific and must not become a global rule candidate by default. If a narrow future exception is needed (e.g., mandatory closing paragraph), it must be separately planned and approved. |

Rationale: `possible_secnav_manual_rule` and `bug_validator_gap` represent corrections that may affect all users or indicate a gap in deterministic validators. They must not be auto-applied but deserve structured logging for future review. Requiring `current_session` scope proves the user wants persistence beyond a single draft.

---

## 3. Ineligible Corrections (Must Be Rejected or Redirected)

| Classification / Condition | Why Ineligible | Redirect Action |
|---|---|---|
| `one_time_wording` | By definition transient | Keep in session scope only |
| `local_command_preference` | Command-specific; belongs in profile, not global rules | Route to Phase C profile promotion if approved |
| `active_draft` only scope | No persistence proven | Require `current_session` scope first |
| `user_rejected=True` | User declined reuse | Exclude permanently |
| `validator_conflict=True` | Contradicts deterministic validator | Block and surface conflict |
| Empty or placeholder value | Not a real candidate | Block |
| `body.*` field paths | Document-specific content; global rules apply to structure, not individual body text | Exclude from Phase D eligibility by default |
| Already logged duplicate | Duplicate candidate | Show existing log entry; allow user to append note instead of re-logging |

## 4. Approval Workflow Before Logging (Mandatory Two-Step Confirmation)

Step 1 — Logging Proposal:

```
PROPOSED GLOBAL RULE CANDIDATE
Candidate ID: <auto-generated>
Type: <possible_secnav_manual_rule | bug_validator_gap>
Field: <field_path>
Corrected value: <sanitized_or_abstracted_value>
Document types: <doc_type(s)>
Component: <component>
User reason: <user_provided_reason>

This will be logged as a candidate for future global rule review.
It will NOT be applied to any other session or user automatically.

Approve logging this candidate? (yes/no/review/edit)
```

Step 2 — Write Confirmation:

```
CANDIDATE LOGGED
Candidate ID: <id>
Field: <field_path>
Logged at: <timestamp>
Status: pending

This candidate will appear in the pending log for future review.
```

Safety rule: No log write occurs unless both steps complete. There is no silent logging and no `--auto-approve` flag in Phase D.

## 6. Protection Against Accidental Public Commit

| Safeguard | How |
|---|---|

| Atomic append | Writes append only; no in-place mutation of existing records |
| No commit hooks | No automation that stages or commits the log file |
| Local path only | Log path resolves to a local file, never a remote URL |
| No real command data | Sanitization rules strip or abstract command-specific values (see Section 8) |
| Validation gate | Pre-write validation rejects records containing likely PII patterns |

## 7. Log Format: JSON Lines (JSONL)

Phase D uses **JSON Lines** (`.jsonl`), one record per line, for these reasons:

- **Append-only friendly** — new records are written as a single line without reading or rewriting the entire file.
- **Stream processing** — review utilities can read line-by-line without loading the full log into memory.
- **Human grep-friendly** — simple `grep` or `jq` one-liners work on individual lines.
- **Resilient to corruption** — a malformed line does not invalidate the entire file.
- **Consistent with Phase A** — session correction JSONL already uses this format.

Each line is a compact JSON object (no internal newlines) to preserve line boundaries. Example:

```json
{"candidate_id":"cand_20260601_abc123","recorded_at":"2026-06-01T12:00:00Z","status":"pending","correction_type":"possible_secnav_manual_rule","field_path":"subj","sanitized_value":"POLICY UPDATE","original_value_sanitized":"POLICY UPDATE.","doc_type":"standard_letter","component":"marine_corps","user_reason":"SECNAV manual says subject should not end with a period","classification_confidence":"high","classification_method":"field_path_keyword","source_correction_id":"corr_abc123","session_id":"sess_20260601_001","review_metadata":null}
```

## 8. Required Fields for Each Pending Candidate Record

| Field | Type | Required | Description |
|---|---|---|---|
| `candidate_id` | string | Yes | UUID or timestamp-random identifier, unique per record |
| `recorded_at` | ISO8601 string | Yes | UTC timestamp when the candidate was logged |
| `status` | enum | Yes | One of: `pending`, `under_review`, `deferred`, `rejected`, `promoted`, `superseded` |
| `correction_type` | string | Yes | `possible_secnav_manual_rule` or `bug_validator_gap` |
| `field_path` | string | Yes | Canonical field path affected |
| `sanitized_value` | string or null | Yes | Abstracted or redacted corrected value (see Section 9) |
| `original_value_sanitized` | string or null | Yes | Abstracted or redacted original system-generated value, for reviewer comparison |
| `doc_type` | string | Yes | Document type context |
| `component` | string | Yes | Service/component context |
| `user_reason` | string | Yes | User-provided explanation, truncated to 500 chars |
| `classification_confidence` | enum | Yes | `high`, `medium`, `low` — based on keyword strength |
| `classification_method` | enum | Yes | `field_path_keyword`, `reason_keyword`, `validator_conflict`, `user_override` |
| `source_correction_id` | string | Yes | Reference to originating session correction ID |
| `session_id` | string | Yes | Session identifier for traceability back to the session JSONL store |
| `review_metadata` | object or null | No | Initially null; populated when status changes (see Section 12) |
| `duplicate_of` | string or null | No | `candidate_id` of earlier duplicate if merged |

## 9. Sanitization and PII Avoidance

Raw corrected values may contain command names, unit identifiers, personnel names, or local addresses. Pending global rule candidates should be **abstracted** where possible so the log stores the *rule implication* rather than the *specific local value*.

### Sanitization Strategy

| Field | Strategy | Example |
|---|---|---|
| `from` | Replace actual command name with `[COMMAND_NAME]`; preserve structural pattern if relevant | `"Commanding Officer, USS NEVERSAIL"` → `"[COMMAND_NAME]"` |
| `to` | Replace actual addressee with `[ADDRESSEE]` | `"Commanding Officer, 1st Marine Division"` → `"[ADDRESSEE]"` |
| `signature` | Abstract to `[SIGNATORY]` | `"J. K. JANICKI"` → `"[SIGNATORY]"` |
| `point_of_contact` | Replace with `[POC_NAME]` and `[POC_CONTACT]` | `"Lt Smith, 703-555-0100"` → `"[POC_NAME], [POC_PHONE]"` |
| `ssic` | Keep if it is a generic/standard SSIC; abstract if command-specific | `"1000"` → `"1000"`; `"N7"` → `"[ORIGINATOR_CODE]"` |
| `originator_code` | Abstract to `[ORIGINATOR_CODE]` | `"N7"` → `"[ORIGINATOR_CODE]"` |
| `unit_identity` | Abstract to `[UNIT_IDENTITY]` | `"NEVERSAIL"` → `"[UNIT_IDENTITY]"` |
| `subj` | Keep verbatim if the rule is about punctuation/structure; abstract if about content | `"POLICY UPDATE."` → `"POLICY UPDATE."`; `"NEVERSAIL POLICY"` → `"[UNIT] POLICY"` |
| `body.*` | **Phase D excluded by default** — document-specific content; if ever revisited, keep structural/language changes; abstract command-specific content | `"The USS NEVERSAIL will..."` → `"The [UNIT] will..."` |
| `ref`, `encl`, `via` | Keep structural changes; abstract specific names | `"Ref (a) USS NEVERSAIL LOI"` → `"Ref (a) [UNIT] LOI"` |
| **All fields** | Never store raw email addresses, phone numbers, or physical addresses | `"j.doe@example.mil"` → `"[EMAIL]"`; `"123 Example Road"` → `"[ADDRESS]"` |
| **All fields** | Never store EDIPI, SSN, DoD ID numbers, or UIC | `"1234567890"` → `"[EDIPI]"`; `"123-45-6789"` → `"[SSN]"`; `"N00123"` → `"[UIC]"` |
| **All fields** | Never store hull numbers, tail numbers, building numbers, or room numbers | `"USS NEVERSAIL (DDG-999)"` → `"[UNIT] ([HULL_NUMBER])"`; `"Bldg 123, Rm 456"` → `"[BUILDING], [ROOM]"` |

### Implementation Approach

A lightweight `_sanitize_value(field_path, raw_value)` helper performs pattern-based abstraction:

1. Run a small dictionary of regex replacements (command name → `[COMMAND_NAME]`, EDIPI → `[EDIPI]`, SSN → `[SSN]`, DoD ID → `[DOD_ID]`, UIC → `[UIC]`, hull number → `[HULL_NUMBER]`, tail number → `[TAIL_NUMBER]`, building number → `[BUILDING]`, room number → `[ROOM]`, email → `[EMAIL]`, phone → `[PHONE]`, address → `[ADDRESS]`).
2. If no known pattern matches, fall back to `[REDACTED_VALUE]`.
3. Never store raw email addresses, phone numbers, physical addresses, EDIPI, SSN, DoD ID numbers, UIC, hull numbers, tail numbers, building numbers, or room numbers.
4. The original raw value remains in the session JSONL store only (already gitignored and local-only).
5. `body.*` field paths are excluded from Phase D eligibility by default; if a future exception is separately approved, the same sanitization rules apply.

If sanitization results in a completely redacted string with no structural information, the candidate should be flagged as `classification_confidence: low` and the user should be warned: "This candidate contains mostly local data. Consider logging as a local command preference instead."

## 10. Distinguishing Candidate Types

### Possible SECNAV Manual Rule

A candidate is classified as `possible_secnav_manual_rule` when:

- The user reason contains keywords: `manual`, `SECNAV`, `M-5216`, `regulation`, `chapter`, `figure`, `rule`, `standard`.
- The field path is `subj`, `ref`, `encl`, `via`, `date`, `classification`, or another structural/formatting field.
- The correction aligns with a known manual convention (e.g., subject line punctuation, reference formatting).

The candidate record should capture:
- The specific manual section or figure referenced by the user (if mentioned).
- Whether the correction contradicts an existing validator finding.
- A suggested rule statement in plain language.

### Validator Bug / Gap

A candidate is classified as `bug_validator_gap` when:

- The user reason contains keywords: `validator`, `should have caught`, `false negative`, `false positive`, `bug`, `missing check`.
- `validator_conflict=True` was set on the correction.
- The correction fixes something the deterministic validator failed to flag, or the validator flagged incorrectly.

The candidate record should capture:
- The validator name or field that failed (if known).
- Whether the gap is a false positive, false negative, or missing rule.
- A suggested validator update description.

### Misclassified Local Command Preference

If a correction was classified as `possible_secnav_manual_rule` or `bug_validator_gap` but is actually a local preference, the user may redirect it:

- Step 1 proposal should include: "This looks like a global rule candidate. If it is actually command-specific, route to profile promotion instead."
- If the user chooses "route to profile", the correction is redirected to Phase C `correction_promote.propose_promotion()` instead of being logged.
- If the user insists on logging anyway, `classification_confidence` is set to `low` and `review_metadata.notes` records the user's override.

## 11. Duplicate Candidate Handling

Before appending a new candidate, the system checks the existing log for duplicates using a fingerprint:

```python
fingerprint = hashlib.sha256(
    json.dumps({
        "field_path": field_path,
        "sanitized_value": sanitized_value,
        "doc_type": doc_type,
        "component": component,
        "correction_type": correction_type,
    }, sort_keys=True).encode("utf-8")
).hexdigest()[:16]
```

| Duplicate Found? | Action |
|---|---|
| No existing match | Append new candidate with `status="pending"` |
| Existing `pending` or `under_review` with same fingerprint | Show the existing `candidate_id`; ask user whether to append a note or skip |
| Existing `rejected` with same fingerprint | Warn user; allow re-logging only with explicit override; set `status="pending"` and link `duplicate_of` to rejected record |
| Existing `promoted` with same fingerprint | Inform user the rule is already promoted; allow note-only append to `review_metadata` |

No automatic deduplication that silently drops records. The user must confirm whether to skip, append a note, or force a new entry.

## 12. Review Lifecycle (Status Transitions)

The `status` field tracks the candidate through its lifecycle:

| Status | Meaning | Who Sets |
|---|---|---|
| `pending` | Newly logged, awaiting review | System on creation |
| `under_review` | A reviewer has opened/claimed this candidate | Reviewer or orchestrator |
| `deferred` | Valid candidate but not actionable now; may be revisited | Reviewer |
| `rejected` | Not a valid global rule or validator gap; reviewer discarded | Reviewer |
| `promoted` | Candidate was promoted to `approved_global_rule` (Phase E+) | Reviewer / promotion utility |
| `superseded` | A later candidate or rule update made this obsolete | System or reviewer |

### Review Metadata Schema

When a status changes from `pending`, the `review_metadata` object is populated:

```json
{
  "reviewed_at": "2026-06-15T10:00:00Z",
  "reviewed_by": "reviewer_id_or_name",
  "review_action": "promoted",
  "review_notes": "Subject punctuation rule confirmed against M-5216.5 Chapter 7, para 3.",
  "linked_rule_id": "rule_v6_cci_subj_punctuation_001",
  "linked_validator_update": "src/cci_subject.py line 42"
}
```

Phase D does **not** implement the review UI or promotion utility. It only defines the schema and ensures `review_metadata` can be stored. The actual review/promotion workflow is Phase E.

## 13. Interaction with Phase C Profile Promotion

Phase D and Phase C are mutually exclusive promotion paths for a single correction:

| Correction Classification | Phase C Path | Phase D Path |
|---|---|---|
| `local_command_preference` | Eligible for profile promotion | **Not eligible** for global candidate log |
| `possible_secnav_manual_rule` | **Not eligible** for profile promotion | Eligible for global candidate log |
| `bug_validator_gap` | **Not eligible** for profile promotion | Eligible for global candidate log |
| `one_time_wording` | Not eligible | Not eligible |

A correction classified as `possible_secnav_manual_rule` or `bug_validator_gap` may be **misclassified** as a local preference. The Phase D proposal should offer:

- "Log as global rule candidate"
- "Route to local command profile instead"
- "Discard"

If the user chooses "Route to local command profile", the correction is passed to `correction_promote.propose_promotion()` with a synthetic `correction_type="local_command_preference"` override, provided it meets Phase C eligibility (field path whitelist, no conflict, etc.).

Conversely, a `local_command_preference` correction should never be accidentally logged to the global pending log. The Phase C promotion UI should not show a "log as global candidate" option.

## 16. What Phase D Must NOT Do

- **No global rule promotion** — Phase D only logs candidates; it does not modify `rules_v6/CCI/`, validators, or prompt contracts.
- **No validator/rule catalog modification** — Do not edit `src/cci_*.py`, `src/validator_runner.py`, or any rule file.
- **No renderer/layout changes** — Do not modify `src/pdf_v6_render.py` or layout profiles.
- **No UI implementation unless separately approved** — No natural-language chat interface beyond the existing orchestrator hooks.
- **No real command/user data committed** — Sanitization must prevent real names, commands, emails, phones, or addresses from entering the log.
- **No automatic pending log creation without explicit approval** — Every log write requires the two-step confirmation workflow.
- **No `--auto-approve` flag** — Not implemented in Phase D.
- **No review/promotion utility** — Status transitions are schema-only; the actual review UI is Phase E.
- **No cross-session candidate logging** — Only current session corrections are eligible.
- **No duplicate auto-merge** — Duplicates are surfaced to the user, not silently merged.

## 18. Open Questions Needing Approval

1. **Fingerprint scope** — Should duplicate detection include `user_reason` text, or only `(field_path, sanitized_value, doc_type, component)`?
2. **Log retention** — Should old `rejected` or `superseded` candidates be pruned after a period, or retained indefinitely for audit?
3. **Review metadata reviewer identity** — Is `reviewed_by` a free-text name, a user ID, or an email? How is identity validated?
4. **Cross-user sharing** — If multiple users on the same machine generate candidates, should they share the same `pending_corrections.jsonl`, or should candidates be namespaced by user?
5. **Notification on duplicate** — When a duplicate is detected, should the system silently append a note, always ask the user, or offer a config toggle?
6. **Misclassification UX** — Should the Phase D proposal offer a one-click "this is actually local" redirect, or require the user to cancel and re-run Phase C separately?
7. **Log size limits** — Should there be a hard or soft cap on `pending_corrections.jsonl` size (e.g., warn at 1 MB)?
8. **Integration with future Phase E** — Should Phase D define a candidate export format (e.g., JSON array) for Phase E review utilities, or should Phase E read the JSONL directly?
